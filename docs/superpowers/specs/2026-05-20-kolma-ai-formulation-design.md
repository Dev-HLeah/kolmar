# 콜마 AI 배합 설계 PoC 디자인

작성일: 2026-05-20

## 1. 배경

이 문서는 `docs/request/RFP By Kolma.pdf`와 `docs/request/Introduce by Kolmar Corp.pdf`를 바탕으로 정리한 PoC 설계안이다.

RFP의 핵심 문제는 다음과 같다.

- 기존 보유 배합비 중 유사한 기초 배합을 찾는 데 시간이 오래 걸린다.
- 배합 설계가 연구원 개인의 경험, 지식, 직관에 의존한다.
- 식품공전, 건강기능식품 기준규격, 원료 사용기준 검토를 수동으로 진행한다.
- 수작업 기반 설계로 오류와 재작업이 자주 발생한다.
- 원료 수급 이슈가 생기면 대체 원료 적용을 위해 재설계와 재검토 시간이 늘어난다.

회사 소개서에서 확인한 콜마비앤에이치의 강점은 고형제와 차별화 제형이다.

- 정제
- 분말
- 츄어블정
- 이중정
- 다발정/초소형 정제
- 쿨멜팅 분말
- 크런키 분말
- 팝핑 분말
- 스틱포
- 멀티PTP
- 분말+액상 프레스샷

따라서 이 서비스는 단순한 범용 배합 추천기가 아니라, 콜마의 처방 자산, 실험 이력, 제형 기술, 포장 옵션, 외부 근거 자료를 함께 활용하는 R&D 의사결정 도구로 설계한다.

## 2. 제품 컨셉

이 서비스는 건강기능식품 R&D 연구자가 기존 처방 지식과 외부 근거 자료를 바탕으로 안전하고 설명 가능한 배합 후보를 빠르게 탐색하는 AI 배합 설계 지원 플랫폼이다.

핵심 목적은 다음과 같다.

- 기존 제품, 원료, 배합 정보를 회사 자산으로 등록하고 조회한다.
- 기존 등록 정보를 기반으로 신규 프로젝트를 시작한다.
- 하나의 제품 개발 프로젝트 안에서 여러 실험 그룹과 try를 관리한다.
- 연구자가 테스트 결과를 기록하고 의미 있는 시도를 마킹한다.
- 유사 처방, 규제, 안전 신호, 특허, 화학 정보, 내부 실험 데이터를 빠르게 참조한다.
- AI가 추천한 후보에 대해 왜 추천했는지, 어떤 위험 신호가 있는지, 어떤 근거가 연결되는지 보여준다.

1차 PoC는 고형제 중심으로 간다.

- 기본 대상: 정제, 분말
- 콜마 특화 제안: 츄어블정, 이중정, 다발정/초소형정제, 쿨멜팅 분말, 크런키 분말, 팝핑 분말, 스틱포, 멀티PTP, 분말+액상 프레스샷 확장 가능성

UI는 화려할 필요가 없다. 연구자가 빠르게 입력하고 비교하고 다시 찾을 수 있는 담백한 업무형 UX를 지향한다.

## 3. PoC 범위

### 포함 범위

- 기존 제품/처방 자산 등록 및 조회
- 기존 제품/처방을 기반으로 신규 프로젝트 시작
- 신규 제품 개발 프로젝트 생성
- 프로젝트 내 실험 그룹 생성
- try 목록 생성 및 관리
- try 정보의 점진적 입력
- 연구자 테스트 결과 등록
- 의미 있는 try 마킹
- 프로젝트별 try 모아보기와 필터링
- 유사 배합 검색
- 규제/안전 신호 기반 경고
- 고형제 제형 적합성 검토
- 콜마 특화 제형/포장 제안
- 외부 근거 데이터 수집 구조
- Open API 호출 결과의 저장 및 재사용
- Supabase 기반 DB, Storage, Auth, pgvector 활용
- 관리자, 연구원, 조회자 권한
- 음성 입력을 입력 방식 중 하나로 설계

### 제외 범위

- 최종 독성 판정
- 최종 규제 적합 판정
- 전자결재 또는 승인 워크플로
- 별도 검토자 역할
- 고객사 포털
- 제조 실행 시스템 연동
- 완전 자동 배합 최적화
- Redis 또는 별도 큐 시스템
- 전용 Vector DB
- Kubernetes
- 온프레미스/프라이빗 클라우드 배포

제외한 항목은 본사업 또는 파일럿 이후 확장 후보로 둔다.

## 4. 핵심 사용자 흐름

### 4.1 제품/처방 자산 관리

기존 제품의 원료 및 배합 정보는 프로젝트의 보조 데이터가 아니라 독립된 핵심 자산으로 관리한다.

연구원은 기존 제품에 대해 다음 정보를 등록하고 조회할 수 있다.

- 제품명
- 제품 카테고리
- 기능성
- 타깃
- 제형
- 포장
- 처방 버전
- 원료 목록
- 원료별 함량, 비율, 단위
- 제조 조건
- 테스트 결과
- 고객사 컨펌 이력
- 생산 이슈
- 규제 검토 이력
- 관련 첨부 파일

기존 등록 제품/처방은 신규 프로젝트의 시작점이 될 수 있다. 기존 처방을 복제해 프로젝트의 기준 try로 만들 때는 원본 제품과 원본 처방 연결 이력을 남긴다.

### 4.2 신규 제품 개발 프로젝트

프로젝트는 하나의 제품을 만들기 위한 개발 단위다.

예시는 다음과 같다.

- 신물억제 제품 개발
- 수면 개선 고형제
- 혈당 관리 분말 제품
- 쿨멜팅 이너뷰티 제품

프로젝트에는 다음 정보를 둔다.

- 목표
- 기능성
- 타깃 고객 또는 소비자
- 희망 제형
- 원가 범위
- 제외 원료
- 참고 제품
- 프로젝트 상태

프로젝트는 다음 방식으로 시작할 수 있다.

- 완전 신규 입력
- 기존 등록 제품/처방 복제
- 여러 기존 처방을 참조 후보로 선택

### 4.3 실험 그룹

프로젝트 안에는 목적별 실험 그룹을 둘 수 있다.

예시는 다음과 같다.

- 신물억제 그룹
- 맛 개선 그룹
- 안정성 개선 그룹
- 원가 절감 그룹
- 제형 적합성 그룹
- 콜마 특화 제형 검토 그룹

그룹은 try가 많아졌을 때 연구자가 실험 목적을 잃지 않게 해주는 단위다.

### 4.4 Try

try는 하나의 실험 가능한 배합 후보 또는 실험 슬롯이다.

예시는 다음과 같다.

- try#1
- try#2
- try#17
- try#100

try는 처음부터 모든 값이 채워진 완성 처방이 아니다. 연구자가 미리 try#1~100을 만들어두고 테스트를 진행하면서 필요한 정보만 채울 수 있어야 한다.

필수값은 최소화한다.

- try 번호
- 소속 프로젝트
- 소속 그룹
- 생성자
- 생성 시각
- 상태

선택값은 다음과 같다.

- 배합 정보
- 원료별 함량
- 단위
- 제형
- 제조 조건
- 원료 변경 이유
- AI 추천 이유
- 규제 체크 결과
- 안전 신호 결과
- 제형 적합성 결과
- 원가 추정
- 연구자 메모
- 테스트 결과
- 첨부 파일
- 근거 링크
- 마킹 사유

try 상태는 다음처럼 둔다.

- `draft`: 번호만 잡힌 초안 또는 부분 입력 상태
- `planned`: 테스트 예정
- `in_progress`: 테스트 중
- `tested`: 테스트 결과 일부 또는 전체 등록
- `candidate`: 의미 있는 후보
- `final_candidate`: 최종 후보
- `discarded`: 폐기
- `on_hold`: 보류 또는 추가 정보 필요

### 4.5 테스트 결과 등록

연구자는 try별로 테스트 결과를 등록한다.

테스트 결과는 정형 입력과 자유 메모를 모두 지원한다.

- 실험 목적
- 실험일
- 실험 조건
- 측정 항목
- 측정값
- 단위
- 결과 판정
- 맛
- 색
- 냄새
- 물성
- 흐름성
- 붕해/용해
- 안정성
- 제조성
- 원가 이슈
- 부작용 의심
- 연구자 코멘트
- 첨부 파일

모든 항목은 필수가 아니다. 연구자가 그 시점에 아는 값만 입력할 수 있어야 한다.

### 4.6 의미 있는 Try 마킹

try가 많이 쌓이면 의미 있는 시도를 빠르게 다시 찾는 기능이 중요하다.

마킹 유형은 다음과 같다.

- 유망
- 문제 발견
- 기준 처방 후보
- 최종 후보
- 폐기
- 재검토 필요

프로젝트 화면에서는 다음 보기를 제공한다.

- 전체 try
- 의미 있는 try만 보기
- 그룹별 try
- 최종 후보 try
- 폐기 try와 폐기 사유
- 특정 원료 또는 함량 변경이 있었던 try
- 테스트 결과, 리스크, 원가, 유사도 기준 정렬

이 기능은 단순 실험 목록을 제품 개발 의사결정 이력으로 바꿔준다.

### 4.7 입력 UX

입력은 연구자가 불편하지 않도록 공통 UX로 묶는다.

지원 입력 방식은 다음과 같다.

- 내부 DB에서 기존 처방 선택
- 표 형태 직접 입력
- 엑셀처럼 복사/붙여넣기
- 원료 자동완성
- 단위 선택 및 변환 보조
- 최근 사용 원료
- 음성 입력 설계

음성 입력은 1차 구현 필수는 아니지만 입력 방식 중 하나로 설계한다.

예시 흐름:

1. 연구자가 "테아닌 200 밀리그램, 아연 8 밀리그램 추가"라고 말한다.
2. 시스템이 음성을 텍스트로 변환한다.
3. 원료명, 함량, 단위를 추출한다.
4. 원료 사전과 매칭한다.
5. 애매한 원료명은 후보로 남긴다.
6. 처방표에 임시 반영한다.
7. 연구자가 확인한 뒤 저장한다.

음성 입력 결과는 바로 확정 저장하지 않는다.

## 5. 데이터 구조

데이터는 장기 자산, 프로젝트 실험, 근거 데이터, 운영 메타데이터로 나눈다.

### 5.1 자산 DB

기존 제품과 처방 정보를 관리한다.

주요 엔티티:

- `Product`
- `ProductFormula`
- `ProductFormulaIngredient`
- `Ingredient`
- `IngredientAlias`
- `DosageForm`
- `ManufacturingConstraint`
- `PackagingOption`
- `ProductTestResult`

역할:

- 기존 제품/처방 자산 관리
- 원료 사전과 별칭 관리
- 유사 배합 검색 기반 제공
- 신규 프로젝트 시작점 제공

### 5.2 프로젝트 DB

신규 개발과 반복 실험 이력을 관리한다.

주요 엔티티:

- `DevelopmentProject`
- `ExperimentGroup`
- `FormulaTry`
- `TryIngredient`
- `TryTestResult`
- `TryMark`
- `TryComparison`

역할:

- 제품 개발 프로젝트 관리
- 그룹별 try 관리
- try별 테스트 결과 축적
- 어떤 배합이 왜 이어졌고 왜 폐기됐는지 추적

### 5.3 근거 DB

추천과 경고의 이유를 저장한다.

주요 엔티티:

- `EvidenceSource`
- `EvidenceItem`
- `EvidenceLink`
- `RegulatoryRule`
- `SafetySignal`
- `ChemicalProfile`
- `RawExternalRecord`
- `NormalizedExternalRecord`
- `SourceSnapshot`
- `DataImportJob`

역할:

- 공식/외부/내부 근거 저장
- Open API 원본 응답 보관
- 정규화 데이터 재사용
- 원료, 제품, 처방, try, 규제 항목과 근거 연결

### 5.4 권한/운영 DB

주요 엔티티:

- `User`
- `Role`
- `Permission`
- `AuditLog`
- `VoiceInputDraft`

역할:

- 관리자/연구원/조회자 권한 관리
- 주요 변경 이력 추적
- 음성 입력 임시 데이터 보관

## 6. 외부 데이터와 재사용 파이프라인

데이터화할 수 있는 것은 최대한 데이터화한다.

외부 API와 문서는 실시간 조회로 끝내지 않고 내부 데이터로 저장 후 재사용한다.

파이프라인:

1. Open API 또는 문서 업로드로 수집한다.
2. 원본 응답 또는 원본 파일을 저장한다.
3. 표준 스키마로 정규화한다.
4. 원료, 제품, 처방, try, 규칙에 매핑한다.
5. 근거 출처와 근거 등급을 부여한다.
6. 필요한 경우 임베딩을 생성한다.
7. 정형 검색 인덱스와 벡터 검색 인덱스를 만든다.
8. 추천, 시뮬레이션, 검색, 리포트에서 재사용한다.

수집 대상:

- 식약처/식품안전나라 데이터
- 건강기능식품 원료 및 제품 데이터
- 법령/고시/기준규격
- KIPRIS 특허 데이터
- PubChem 화학 데이터
- 내부 제품/처방/테스트 데이터
- 고객 실험 데이터
- 업로드된 PDF, 엑셀, 테스트 첨부 파일

근거 등급:

- 공식 규제/고시
- 공식 공공 데이터
- 내부 실험/품질 데이터
- 고객사 컨펌 데이터
- 특허
- 논문/문헌
- PubChem/화학 데이터
- AI 추론

화면에서는 공식 기준, 참고 근거, AI 추론을 명확히 구분해야 한다.

## 7. Vector 검색

Vector DB는 적극 활용하되 원본 저장소로 쓰지 않는다.

정형 데이터의 원본은 Supabase Postgres에 둔다. 파일은 Supabase Storage에 둔다. 벡터 임베딩은 의미 기반 검색을 위한 인덱스로 쓴다.

사용 예시는 다음과 같다.

- 특허/문서 의미 검색
- 내부 실험 메모 의미 검색
- 유사 목적의 과거 try 검색
- 원료명 별칭과 표현 차이 보완
- 자연어 질문 기반 검색
- RAG 방식의 근거 요약

질문 예시:

- "신물억제 관련 과거 실패 사례 찾아줘"
- "이 원료 조합과 비슷한 실험 결과 찾아줘"
- "쿨멜팅 분말과 비타민 C 관련 근거 찾아줘"

PoC에서는 Supabase Postgres의 pgvector를 사용한다. 데이터 규모나 성능 문제가 커질 때만 전용 Vector DB를 검토한다.

## 8. 추천/시뮬레이션 구조

추천 엔진은 한 번에 정답 처방을 내는 구조가 아니다. 여러 후보 try를 만들고, 각 후보의 이유, 근거, 리스크, 제형 적합성을 함께 보여준다.

흐름:

1. 입력 조건 또는 기준 처방을 받는다.
2. 유사 기존 처방을 찾는다.
3. 관련 근거를 찾는다.
4. 후보 try를 생성한다.
5. 규칙 기반 규제 체크를 수행한다.
6. 안전 신호를 확인한다.
7. 제형 적합성을 평가한다.
8. 콜마 특화 제형/포장 옵션을 제안한다.
9. 연구자가 확인 가능한 형태로 결과를 보여준다.

후보 유형:

- 안정성 우선 후보
- 기능성 우선 후보
- 맛/관능 개선 후보
- 원가 절감 후보
- 콜마 특화 제형 후보
- 기존 처방 최소 변경 후보

규제/안전 시뮬레이션은 1차 PoC에서 확정 판정이 아니라 위험 신호 안내로 둔다.

예시:

- 일일섭취량이 알려진 범위를 초과할 가능성
- 섭취 시 주의사항 존재
- 특정 원료 조합에서 과거 내부 이슈 존재
- 유사 특허/문헌에서 주의 신호 확인
- PubChem 또는 화학 데이터상 추가 검토 필요

시스템은 공식 근거가 없는 한 "독성이 있다"고 단정하지 않는다. "이런 근거 때문에 추가 검토가 필요하다"는 방식으로 안내한다.

## 9. 제형 및 콜마 특화 제안

PoC는 정제와 분말 중심이지만, 콜마의 특화 제형을 제안 레이어로 포함한다.

평가 대상:

- 정제 가능성
- 분말화 가능성
- 츄어블정 가능성
- 이중정 가능성
- 초소형정제 가능성
- 쿨멜팅 분말 가능성
- 크런키 분말 가능성
- 팝핑 분말 가능성
- 스틱포 가능성
- 멀티PTP 가능성
- 프레스샷 향후 확장 가능성

검토 신호:

- 흐름성
- 흡습성
- 맛 마스킹 필요성
- 색 마스킹 필요성
- 냄새 마스킹 필요성
- 안정성 리스크
- 붕해/용해 리스크
- 제조 제약
- 포장 적합성

이 결과는 최종 제조 판단이 아니라 초기 제안과 연구 검토를 돕는 정보다.

## 10. 권한

PoC 역할은 세 개만 둔다.

### 관리자

가능한 일:

- 사용자/권한 관리
- 외부 데이터 수집 상태 확인
- 원료명 매핑 수정
- 마스터 데이터 관리
- 주요 운영 로그 조회

### 연구원

가능한 일:

- 제품/처방 등록
- 프로젝트 생성
- 실험 그룹 생성/수정
- try 생성/수정
- 테스트 결과 등록
- 의미 있는 try 마킹
- 추천/검색 기능 사용

### 조회자

가능한 일:

- 허용된 제품 조회
- 허용된 프로젝트 조회
- 처방, try, 근거, 리포트 조회

조회자는 수정할 수 없다.

PoC에는 별도 검토자 역할과 승인 워크플로를 넣지 않는다. 연구원이 직접 최종 후보, 폐기, 재검토 필요 등을 마킹한다. 실제 내부 업무 방식이 확인되면 이후 검토자/승인 기능을 추가한다.

## 11. 감사 로그

PoC에서는 핵심 변경 이력만 남긴다.

로그 대상:

- 제품/처방 생성 및 수정
- 프로젝트/try 생성 및 수정
- 테스트 결과 등록
- try 마킹 변경
- 기존 처방을 프로젝트로 복제
- 외부 데이터 수집
- 원료명 매핑 수정
- 권한 변경
- 음성 입력 초안 확정

목적은 무거운 결재가 아니라 추적성 확보다.

## 12. PoC 인프라/기술 스택

완제품이 아니라 PoC이므로 가볍게 간다.

프론트엔드:

- React.js
- TypeScript
- Vite
- Vercel 배포

백엔드:

- NestJS
- Prisma ORM
- Railway 배포

DB/스토리지/Auth/Vector:

- Supabase Postgres
- Supabase Storage
- Supabase Auth는 선택적 인증 기반으로 사용
- Supabase pgvector로 의미 검색 지원

PoC에서 사용하지 않는 것:

- Redis
- 별도 큐 시스템
- 전용 Vector DB
- Kubernetes
- 프라이빗 클라우드

외부 데이터 수집:

- `DataImportJob` 테이블로 수집 상태를 추적한다.
- Railway Cron Jobs 또는 NestJS command script로 주기 수집한다.
- Open API 응답과 정규화 결과를 Supabase에 저장한다.
- 저장된 데이터를 검색, 추천, 리포트에서 재사용한다.

이 구조는 빠른 PoC 개발에 맞고, 필요 시 본사업 단계에서 확장할 수 있다.

## 13. 확장 경로

PoC 이후 필요하면 다음을 추가한다.

- Redis 또는 큐 시스템
- 전용 Vector DB
- 검토자 역할과 승인 워크플로
- 고객사 조회 포털
- SSO
- 프라이빗 클라우드 또는 온프레미스 배포
- 정교한 독성/상호작용 예측 모델
- 제조 시스템 연동
- 자동 리포트 내보내기
- 서버 기반 음성 인식

## 14. 성공 기준

PoC가 성공했다고 볼 수 있는 기준은 다음과 같다.

- 기존 제품/처방을 등록하고 조회할 수 있다.
- 기존 제품/처방을 기반으로 신규 프로젝트를 시작할 수 있다.
- 프로젝트 안에서 실험 그룹과 여러 try를 관리할 수 있다.
- try 정보는 필수값만으로 생성 가능하고, 나중에 선택값을 채울 수 있다.
- try별 테스트 결과를 등록할 수 있다.
- 의미 있는 try를 마킹하고 프로젝트별로 모아볼 수 있다.
- 유사 처방과 관련 근거를 검색할 수 있다.
- Open API 데이터를 저장하고 재사용할 수 있다.
- Vector 검색으로 관련 문서, 메모, 처방을 찾을 수 있다.
- 추천 결과에 이유, 리스크, 근거 링크가 함께 표시된다.
- 콜마 특화 제형/포장 제안이 결과에 포함된다.
- UI가 담백하고 연구 업무에 방해되지 않는다.
